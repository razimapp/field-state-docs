Customer view
MVP decision-intelligence product

Fieldstate helps football people compare decisions.

The MVP answers one practical question: if this player made a different decision here, what might the next phase of play look like?

Fieldstate turns real match moments into clear what-if decision analysis.
Brand principle

Calm, practical, credible

The product should help analysts and coaches discuss decisions without pretending to predict a perfect alternate reality.

What customers can do

1

Pick a real event

Select a pass or carry from a real match and see the original context.

2

Edit the action

Change the pass target or carry endpoint to test a plausible alternative.

3

Compare the trade-off

Review possession value, chance creation, turnover risk, plausibility, and explanation.

Why it matters

For analysts

  • Review key decisions faster
  • Compare safe and progressive options
  • Turn match data into scenario analysis
  • Support coaching clips with clear evidence

For coaches and academies

  • Show players why one option may be better
  • Explain risk and reward in simple terms
  • Build a library of decision scenarios
  • Spot recurring decision-making patterns

The MVP does not replay the whole match or guarantee what would have happened. It helps football people understand the trade-off between one decision and another.

Football What-If Simulation MVP

A first-pass MVP to test whether a football event can be adjusted and followed by a plausible alternative next phase of play.

Working principle: use AI to accelerate research, explanation, and documentation, but keep the football outcome grounded in data, code, and validation.
Objective

Show what could happen next

The MVP lets a person select one event from a real football match, adjust a pass or carry, and compare the original decision against a plausible alternative.

Scope

Keep the first version narrow

The first version should simulate the next phase, not claim to replay the whole match or predict a perfect alternate reality.

MVP Scope

Included in the first pass

  • StatsBomb Open Data loading, with source attribution
  • Real match and event selection
  • Pass or carry edit flow
  • Original versus counterfactual comparison
  • Basic next-phase simulation
  • 2D pitch visualisation
  • Action-value output using xT / VAEP-style logic
  • Claude/Fable explanation layer for analyst-readable summaries
  • Short feasibility report

Excluded from the first pass

  • Full-match alternate reality simulation
  • Paid data integration
  • Elite-club-grade player-style modelling
  • Full tracking-data realism
  • Large multi-user SaaS platform
  • Transformer training from day one
  • Fable-generated football outcomes treated as truth
  • Commercial accuracy claims

Build Principles

  • Model first: metrics come from event data, xT / VAEP-style logic, and tested code.
  • Fable second: Fable suggests factors, refines documents, and explains model output.
  • Python first: data loading, model features, and feasibility should move quickly before heavier product infrastructure.
  • Cost control: prompt caching keeps the stable project context reusable across research and explanation calls.

30 Working Day Plan

Week 1

Load data, expose a small Python API, show matches, and select events.

Week 2

Add xT / possession-value proxies, plausibility checks, and pass/carry edits.

Week 3

Build baseline branch comparison and structured scenario outputs.

Week 4-6

Add Fable explanations, polish visualisation, test plausibility, and prepare the demo.

Success for the first pass means the output is understandable, plausible, and useful enough to justify deeper data access and model investment. If the core scope is completed in fewer than 30 working days, remaining time will be used to improve the demo, add additional scenario factors, refine model outputs, strengthen the visual experience, and iterate based on investor or analyst feedback.

Use of Anthropic Fable

Where Fable belongs

Fable is best used for deep research, document refinement, candidate factor generation, evaluation design, and analyst-readable explanations of model output.

Where Fable does not belong

Fable should not be the core football simulation model. The outcome path should come from event data, deterministic code, statistical models, and validation.

Prompt caching

Cache the stable project brief, modelling assumptions, schemas, and tone rules. Send only the current event, document section, or research question as the changing input.

Output discipline

Use structured JSON for scenario explanations, factor libraries, and document refinements so the product can parse, review, and audit AI output.
